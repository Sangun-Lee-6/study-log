# **실무적인 데이터 모델링 프로세스**

> 참고자료 : https://joshua-data.medium.com/my-analytics-engineering-process-ko-833684c87e77

- 실무에서 모델링을 효율적으로 하려면 Grain과 최종 형태를 먼저 확정하기
- 데이터 모델링은 SQL 작성 기술이 아니라, 원천 데이터를 이해하고 비즈니스가 원하는 형태로 안전하게 전달하는 설계 작업

- 데이터 모델링 순서

## 1. 원천 데이터를 뜯어보기

- 한 행이 무엇을 의미하는지, 어떤 컬럼이 어떤 컬럼에 종속되는지, 값이 어떻게 쌓이는지 확인하기
- 여기서 확인해야할 것
- [ ] 테이블 간 관계: 1:1인지, 1:N인지, N:M인지
- [ ] 컬럼 의존성: 예를 들어 `type_id`가 있으면 `type_name`이 사실상 따라오는 값인지
- [ ] 값 분포: distinct 값, null 비율, 이상치
- [ ] Grain: 이 테이블의 1행이 정확히 무엇인지
- [ ] 도메인 흐름: 데이터가 실제 비즈니스에서 어떻게 생성되고 쌓이는지
- 이를 확인하지 않고 바로 SQL부터 작성하면, 나중에 조인 중복⋅집계 기준 불명확으로 다시 처음으로 돌아가게 됨
- 모델링은 원천 데이터 이해에서 시작함
- 실무적으로 이렇게 생각하기
- [ ] “이 테이블 1행은 주문 1건인가?”
- [ ] “아니면 주문상품 1건인가?”
- [ ] “아니면 사용자의 하루 행동 요약 1건인가?”

---

## 2. 그 다음에 무엇을 만들 것인가 정하기

- 원천 데이터를 이해하고 바로 Fact, Dim을 만들지 않고 최종 소비 형태(Target Shape)을 정해야함
- 최종 형태 예시
  - Fact/Dim 스타 스키마 또는 OBT
  - AU(활성 사용자) 테이블
  - Aggregate Table
  - 백엔드 서빙용 집계 테이블
- 모델링은 누가 어떻게 쓸지에 맞춰 최종 소비 형태를 정하기
  - 분석가가 다양한 슬라이싱을 해야 하면 → Fact/Dim 또는 OBT
  - 대시보드가 고정된 지표 위주면 → Aggregate Table
  - 제품 API가 바로 읽어가야 하면 → 백엔드 서빙용 테이블
  - 사용자 활동 분석이 핵심이면 → user/day 수준 AU 테이블

---

## 3. 가장 중요한 단계 : Fact의 Grain을 확정하기

- Fact 테이블의 Granularity를 반드시 확정하라
- 예: `date x user_id` 인지, `order_id x item_id` 인지. 이게 이후 모든 쿼리 로직의 기준점이 됨
- “나는 무엇을 한 줄로 셀 것인가?”
- “이 팩트의 최소 측정 단위는 무엇인가?”
- 예시
  - `date x stock_code` → 종목별 일별 지표
  - `date_hour x stock_code` → 종목별 시간 단위 지표
  - `news_id` → 뉴스 1건 단위
  - `disclosure_id` → 공시 1건 단위
- 이걸 먼저 정하지 않으면 결국 중복, 누락 팬아웃이 생김

---

## 4. 아직 SQL 쓰지 말고, 데이터 흐름부터 그리기

- 최종 레이어까지 가는 경로를 먼저 설계해보기
- 즉, 어떤 Fact 소스를 어떤 키로 조인하고, Dim은 어느 시점에 붙이고, 어떤 필터와 집계가 필요한지 화이트보드 수준으로 먼저 설계
- 다음을 생각해보기
  - Fact 후보 테이블은 무엇인가
  - 조인 키는 무엇인가
  - 어느 테이블이 기준 테이블인가
  - 어디서 중복 위험이 생기는가
  - 어떤 시점 컬럼을 기준으로 붙일 것인가
  - 최종 출력 컬럼은 무엇인가

---

## 5. Silver는 중간 결과 저장소가 아니라 재사용 가능한 표준 레이어

- 중간 로직을 그냥 물리화하지 말고, 재사용성과 확장성을 기준으로 결정하라는 것
- Silver로 빼야 하는 경우
  - 다른 마트에서도 재사용될 가능성이 높다
  - 동일한 정제/조인 로직이 여러 번 반복된다
- 굳이 Silver로 빼지 말아야 하는 경우
  - 이 마트에서만 1회성으로 쓰인다
  - 오히려 테이블이 늘어나서 복잡도만 증가한다
- 애매한 Silver 테이블이 많아진다면 조직 전체가 같은 역할의 비슷한 테이블을 여러 개 갖게 됨
  - 그럼 나중에는 어느 테이블이 표준인지 아무도 모르게 됨
- Silver는 “편해서 일단 저장”이 아니라 “조직 내 표준 중간 산출물로 계속 살아남을 가치가 있는가”로 판단

---

## 6. SQL 작성 단계에서는 한 번에 잘 짜는 것보다 안전하게 짜는 것이 더 중요

- 출력 형태와 증분 전략을 먼저 정하기
  - Full refresh인지
  - Incremental인지
  - Incremental이면 멱등성이 보장되는지
  - 특히 이는 Airflow/dbt 환경에서는 특히 중요함, Dag가 여러 번 돌아도 같은 결과가 나와야 하므로
- 전처리를 초반에 강하기 수행
  - 첫 CTE에서 `TRIM`, `NULLIF`, `CAST`, `COALESCE`, `LOWER` 등으로 정제
  - 지저분한 원천 데이터를 아래 단계로 내려가면 쿼리가 복잡해짐
- CTE는 단일 목적이어야 함
  - 하나의 CTE에서 조인, 필터, 집계, 피벗을 다 하지 말라는 의미
  - `base_*`: 컬럼명 통일, 타입 정리
  - `filtered_*`: 범위/조건 필터링
  - `joined_*`: 필요한 차원 결합
  - `agg_*`: 집계
  - `final`: 최종 출력
  - 이렇게 해야 디버깅이 쉬움
- 컬럼명은 초반에 통일
  - 여기저기 `user_id`, `user_idx`, `member_id`가 섞여 있으면 조인 실수 발생

---

## 7. Fact와 Dim 조인은 가장 위험한 구간

- Dim 조인 키가 유니크한지 반드시 확인할 것
- 그렇지 않으면 Fact 행이 복제되어 Fan-out이 발생함
- 실무에서 자주 발생하는 문제(예시)
  - fact_stock_price를 `stock_code`로 dim_stock에 조인했는데
  - dim_stock에 동일 `stock_code`가 여러 행 있으면
  - 가격 1행이 2행, 3행으로 늘어남
  - 그 결과로 거래량 합계가 갑자기 2배⋅건수 집계가 예상보다 큼⋅대시보드 숫자가 틀림
- 그래서 조인 전에는 항상 확인해야 함
  - dim의 조인 키가 truly unique한가
  - SCD Type 2처럼 이력 테이블은 아닌가
  - 최신 1건만 남겨야 하는가
  - null 키는 어떻게 처리할 것인가

## 8. 모델링 이후 점검 단계

1. 네이밍

- 사람이 읽어도 이해돼야 하고, 나중에 비슷한 테이블이 생겨도 헷갈리지 않아야 함

1. 카탈로그

- Grain, PK, 컬럼 의미, 사용 목적을 짧고 정확하게 남기기

1. 데이터 품질 검증 6가지
   - Completeness
   - Uniqueness
   - Validity
   - Timeliness
   - Accuracy
   - Consistency

## 이 글에서 강조하는 실무적인 포인트 3개

1. Grain을 먼저 확정하기

- 이게 모델링의 기준점
- 대부분의 문제는 “조인 실수”처럼 보이지만, 실제로는 Grain이 불명확해서 생김

2. Silver는 확장성을 기준으로 설계

- 중간 테이블을 많이 만드는 게 좋은게 아니라 재사용 가능하고 조직 표준이 될 수 있는 것만 남겨야 함

3. 중복 제거를 반복하고 있다면 업스트림 설계가 잘못된 것

- `QUALIFY ROW_NUMBER() = 1` 같은 중복 제거가 자주 나온다면, downstream에서 봉합 중인 것
- 근본 원인은 위쪽 레이어의 PK/Grain 불명확성일 가능성이 높음
