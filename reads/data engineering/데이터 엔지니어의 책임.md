# 데이터 엔지니어의 책임

# 1. 데이터 엔지니어의 책임에 대한 인식

### 데이터 엔지니어의 책임과 가치

🔹 데이터 엔지니어의 책임을 모른다면 데이터 엔지니어로서 가치를 창출하기 어려움

- PoC, 파이프라인 코드 작성 등 업무를 왜 하고 있는지 모르면서 일하게 됨
- 주어진 작업을 완료하는 것을 가치로 여길 수 있음
- 따라서 데엔의 책임 범위를 알아야함

### 데이터 엔지니어의 책임 범위

🔹 데엔의 책임 범위

- <견고한 데이터 엔지니어링>
- 데이터 엔지니어링 : 원시 데이터로 고품질의 일관된 정보를 생성하고 분석과 같은 유즈케이스를 지원하는 시스템을 개발, 구현 및 유지 관리
- 데이터 엔지니어 : 데이터를 가져와서 데이터 유즈케이스(ex. 분석)에 데이터를 제공하는 데이터 엔지니어링 수명주기를 관리

🔹 데엔의 책임이 갖춰져야 데엔 업무의 가치 창출 기준이 됨

- 무엇이 중요한지, 문제를 파악하고 예방하는 법, 무엇을/왜 배워야하는지

🔹 <견고한 데이터 엔지니어링>의 1장, 2장을 읽고 데엔의 책임에 대해 확실히 이해하는 것을 추천

- 단순히 도구나 개념을 맹목적으로 배우는 것보다 더 큰 동기를 부여받을 수 있을 것

# 2. 모든 회사가 빅데이터를 보유하고 있는 것은 아니다

### 데이터 엔지니어의 가치는 기술이 아니라 회사의 비즈니스에 맞춰진다

🔹 데이터 엔지니어의 가치는 회사의 비즈니스를 위해 데이터를 관리하는 것

- 데엔의 가치는 빅테크와 같은 기술⋅운영이 아니라 회사의 비즈니스를 위해 데이터를 관리하는 것
- 모든 회사가 구글 같은 빅테크처럼 운영되지 않음
- 각 회사마다 필요한 데이터 활용 능력과 처리해야 할 데이터 양은 비즈니스에 따라 다름
- 회사가 원하는 방식으로 데이터를 활용할 수 있는 것이 중요함 → 15분만에 10TB 데이터를 처리하는 것은 관심이 없음

### 빅데이터 환경에서만 데이터 엔지니어가 성장하는게 아니다

🔹 빅데이터 환경에서만 데엔이 성장하는게 아님, 해결해야 할 문제는 많음

- ex. 테이블의 특정 데이터 공백 이슈, 데이터 모델링 부재, DA가 데이터를 찾기 어려운 이슈

🔹 데이터 엔지니어의 가치는 회사의 비즈니스를 위한 데이터

- 데엔의 가치가 반드시 고급 기술을 다루거나 빅데이터를 처리하는 것을 의미하지 않음
- 데엔의 가치는 원천 시스템에서 데이터를 추출하고, 안정적으로 저장 및 관리하며, 필요할 때 누구나 데이터를 원활하게 사용할 수 있도록 지원하는 일
- 이러한 일은 조직의 상황⋅요구사항에 맞는 도구, 솔루션으로 진행해야함
- 이렇게 생각해야 회사의 규모와 상관없이 데엔의 가치에 집중할 수 있음

# 3. 기술적 능력

### 기술적인 능력을 키우기 위해 필요한 3가지

🔹 기술적인 능력을 키우기 위해 필요한 3가지 조건

- 기초부터 탄탄히 이해하기
- 무엇이 더 올바른 접근인지 인지할 것
- 제대로 하고 있는지 계속 확인할 수 있는 피드백 루프를 만들 것

🔹 운영 업무를 위해 기초가 필요함

- 구현과 운영은 다름
- 기초를 이해하면 사용하는 기술의 효율적인 관리⋅버그 조기 감지 및 해결⋅더 현명한 결정⋅더 빨리 학습 가능
- 기초를 이해하면서 학습하면 시간이 더 오래 걸리고, 이력서에 기술을 추가하는데 오래 걸리겠지만 그럴만한 가치가 있음

### 기초부터 탄탄히 이해하기

🔹 기초를 이해하기 위한 질문 : X는 어떤 문제를 해결하는가?(X : 도구, 기술, 서비스)

- 답변이 자세할 수록 기초를 더 빨리 이해할 수 있음
- ex. Spark는 어떤 문제를 해결하는가?
- 효율적인 분산 데이터 처리 → MapReduce가 가능함
- MapReduce 작업 작성이 직관적이지 않았다 → Spark는 더 친숙한 API를 제공함 → 여기서 DataFrame, Dataset에 대해 더 자세히 알아볼 수 있음
- MapReduce는 느림 → Spark는 데이터를 메모리에서 처리함 → RDD와 셔플링에 대해 저 자세히 알아볼 수 있음
- Spark는 MapReduce → 데이터 계보처럼 내결함성도 보장해야 함
- MapReduce는 사용 사례가 제한적입니다 → Spark는 머신러닝, 실시간 처리 등을 지원

🔹 이론에서 끝나지 말고 직접 실습으로 배운 내용을 검증하기

🔹 공통적인 기본 원리를 파악하기

- Snowflake를 배우다 보면 컴퓨팅과 스토리지가 내부적으로 분리되어 있다는 것을 알게됨
- 이런 패턴은 Databricks와 BigQuery에도 있다는걸 알면, 각 시스템의 차이점만 파악하면 되므로 학습 속도도 높일 수 있음
- 기초에 집중할수록 기술 간 공통점을 많이 알게 되고, 더 넓은 시야를 갖게 되어 편견 없이 시스템을 이해할 수 있음

### 무엇이 더 올바른 접근인지 인지할 것

🔹 어떤 일을 하는 것에 대한 올바른 방법을 아는 것

- 파이프라인 코딩, 클러스터 관리, SQL 작성 등 모든 작업에는 항상 모범 사례가 존재함
- 설명서, 회사 표준에 제시된 모범 사례를 따를 것, 이러면 시간을 절약하고 사용하는 도구를 최대한 활용하며 더 쉽게 협업 가능

### 제대로 하고 있는지 계속 확인할 수 있는 피드백 루프를 만들 것

🔹 올바른 방향으로 가고 있는지 확인하기 위해 피드백 루프는 필수적

- 혼자 학습하더라도 학습 과정을 고립시키지 말 것
- 이를 통해 더 빠르고 효율적으로 학습
- AI에게 피드백을 요청하는 것도 괜찮음
- 업무 경험이 많은 사람들이 있다면 피드백을 요청
- 인터넷에 학습 내용을 공유하고, 다른 강의/프로그램도 수강하고 포럼에도 참여해서 피드백

# 4. 기술적인 능력만으로는 멀리 갈 수 없다

### 기술적인 능력만으로는 부족하다

🔹 멀리 가려면 기술적인 능력 외에 문제 해결 능력 및 소통 능력이 필요하다

- 단순히 복잡한 파이프라인을 구축한다고 주니어에서 시니어, 매니저가 되는게 아님
- 조직에 긍정적인 영향을 미치기 때문에 가능한 것
- 긍정적인 영향을 위해선 기술적인 능력에다가 문제 해결 능력, 소통 능력이 필요함

### 문제 해결 능력

🔹 정확히 표현하면 문제를 다루는 능력

- 문제 식별하기, 더 작은 문제로 나누기, 우선 순위 정하기, 현재 제약과 상황을 고려해 효율적으로 해결하기

🔹 문제를 사전에 해결하는 능력이 성장에 도움

- 경력 초반에는 명확하게 정의된 문제가 많지만, 이런 문제가 생기기를 기다리는 것은 성장에 도움이 안됨
- 명확하게 정의된 문제 예시 : 측정 가능한 결과를 갖는 작업, 데이터 불일치를 감지하고 해결하기 등
- 사전에 문제를 해결하기 위해선 비즈니스 용어와 기술 용어를 모두 이해해야함
- 그래야 문제를 명확하게 정의하고 이를 기술적인 세부 사항으로 옮길 수 있음

🔹 막연한 문제를 해결하기 위한 출발점

- 영향력이 큰 문제는 대부분 막연함
- 범위가 넓고, 여러 사람이 얽혀 있고, 서로 다른 분야의 지식이 필요하고, 영향을 주는 변수가 많음
- 『Bulletproof Problem Solving』에 나오는 7단계 프레임워크는 이러한 막연함을 해결하는 좋은 출발점

🔹 평소에 문제를 먼저 발견하고 해결해서 가치를 만들어내는 태도도 중요함

- 처음부터 거대한 문제를 해결할 필요 없이, 주변의 문제를 체계적으로 다뤄보기
- 시니어들이 문제를 해결하는 방식을 주도적으로 배워도 문제 해결력은 길러질 수 있음

### 소통 능력

🔹 문제 해결을 위해 소통 능력도 필요함

- 데엔은 문제를 해결할 때 혼자서 해결하는 경우는 거의 없고 여러 사람과 협업이 필요
- 효과적인 협업을 위해 소통하는 방법을 배워야함

🔹 소통은 양방향적

- 입력 : 여러 사람으로부터 정보를 수집하고 분석
- 출력 : 다양한 형태(말하기, 쓰기 등)로 무엇을 하고 있는지⋅무엇을 할 것인지를 명확하게 표현해야 함

🔹 소통 능력을 기르기 위한 방법

- 우선 다른 사람의 의견을 잘 듣고 상대방의 말에 집중하고 이해되지 않으면 질문하기
- 자신의 생각을 명확하게 표현하려고 노력하고 문서 작성에도 신경을 쓰기
- 정확한 용어를 사용하는 것도 중요함
