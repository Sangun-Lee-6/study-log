# **Kimball Dimensional Modeling Overview**

# 1. Intro

빅데이터 저장과 처리를 수행하는 다양한 기술과 도구가 이미 존재하는 시대임

과거에는 HW가 비쌌고, 견고한 데이터 인프라에는 구축에 시간이 오래 걸렸으므로, 데이터를 시스템에 넣기 전에 데이터 모델링을 신중하게 진행했음

최근에는 빠른 결과물을 위해, 또 리소스를 더 넣으면 어떻게든 해결될 것이라는 이유로 데이터 모델링을 가볍게 생각함

그러나 데이터 모델링은 여전히 중요함

왜냐하면 데이터 모델링은 단순히 저장 형식을 정하는 작업이 아니라, 데이터를 일관된 의미와 구조로 조직해 사람이 신뢰할 수 있게 해석하고 시스템이 안정적으로 활용하도록 만드는 기반이기 때문

# 2. Data Warehousing

🔹 빌 인먼의 데이터 웨어하우징 : 데이터 생성 시스템과 분석 시스템을 분리하는 것이 표준이 됨

- 데이터 생성 시스템은 OLTP를 사용(높은 동시성, 트랜잭션 조회)
- 분석 시스템은 데이터 생성 시스템으로부터 정보를 수집하고 정리, OLAP 사용(고성능 질의에 최적화, 높은 동시성까지는 요구 X)
- 이 글은 분석 시스템, 즉 데이터 웨어하우징에 초점을 맞춤

🔹 데이터 웨어하우징 시스템의 특성

- [ ] 비즈니스 사용자에게도 직관적
- [ ] 다양한 데이터는 일관된 레이블과 정의로 제공
- [ ] 요구사항과 변화에 적응 가능
- [ ] 민감한 정보는 보호
- [ ] 의사결정을 지원할 수 있는 적절한 데이터가 존재
- [ ] 비즈니스 사용자가 DW/BI 시스템을 받아들여야 함(아무도 사용하지 않으면 의미 없음)

🔹 킴볼은 차원 모델링이 이 기준을 충족하는 DW 솔루션에 도움이 된다고 봄

# 3. Dimensional Modeling

### 차원 모델링

🔹 차원 모델링의 등장

- 차원 모델링은 1996년에 출간된 Ralph Kimball의 저서 _The Data Warehouse Toolkit_ 1판에서 처음 등장

🔹 차원 모델링이란

- 이 접근법은 단순함을 지향 → 이는 대부분 비즈니스 사용자가 직관적으로 사고하는 방식는 방식
- 비즈니스 사용자는 측정 가능한 지표 & 그 지표가 관찰되는 맥락으로 이해함
  - ex. 판매 매니저는 제품 카테고리별⋅여러 지역에 걸쳐⋅시간의 흐름에 따라 매출 성과를 분석
  - 이러한 사고 방식은 차원적
- 제품⋅지역⋅시간은 모두 차원(성과를 평가할 수 있는 다른 관점)
- 차원 모델링을 통해 사용자가 데이터를 구체적으로 이해할 수 있는 틀을 제공
- 데이터 구조도 기술 중심이 아닌 직관적이고 이해하기 쉽게 설계됨

### Star Schema

🔹 정규화

- 정규화의 목표 : 중복 제거로 데이터 무결성 보장
  - 데이터를 많은 엔티티로 나누고 각 엔티티를 관계형 테이블로 구성
  - 데이터 무결성이 우선인 운영 처리 환경에서 유용

🔹 정규화와 차원 모델링

- DW에는 정규화는 너무 복잡함
- 수백 개의 엔티티가 있는 ERD를 보면서 분석을 하기 어려움
- 보통 차원 모델을 스타 스키마 형태로 구현함
- 중앙의 팩트 테이블을 여러 개의 차원 테이블이 둘러싸는 형태

### Fact

🔹 팩트 테이블

- 스타 스키마의 중심 테이블
- 조직의 비즈니스 프로세스로부터 발생하는 성과 측정값을 저장
- 킴볼은 더 높은 유연성을 위해 가장 낮은 수준의 측정값을 저장하도록 권장
  - ex. 미리 월별 합계로 저장하지 않고, “2026-04-13 10:03에 고객 A가 상품 1개를 12,000원에 구매했다”처럼 실제 발생한 개별 거래 단위 그대로 저장한 값

🔹 Grain

- 팩트 테이블의 각 행은 하나의 측정 이벤트에 대응
- 각 행의 데이터는 특정한 상세 수준을 가짐(grain)
- 하나의 팩트 테이블 안의 모든 행은 동일한 grain을 가져야 함
- ex. 이벤트 추적용 팩트 테이블에서는 모든 행은 사용자 이벤트 라는 수준에 대응

🔹 팩트 테이블의 데이터 구성

- FK : 관련 차원 테이블로 연결되는 키
- 측정값 : 수치 값
- 팩트 테이블의 모든 키가 해당 차원 테이블의 기본 키와 정확히 일치하는 경우 → 이 테이블은 참조 무결성을 만족

### Dimension

🔹 차원 테이블

- 차원 테이블은 팩트에 대한 설명을 제공
- 팩트에 대한 누가, 무엇을, 어디서, 언제, 어떻게, 왜에 대한 정보
- 각 차원 테이블은 하나의 비즈니스 차원에 초점을 맞춤
- 차원 테이블은 측정값(팩트)에 맥락을 제공하므로 DW 시스템에서 핵심적인 역할
  - ex. 매출 수치값이 커졌다는 사실만으로는 비즈니스 인사이트를 얻기 어려움
- DW 품질은 결국 차원의 품질에 달려있으므로 차원 테이블의 속성(컬럼)은 최대한 비즈니스 용어와 가깝게 모델링해야 함

### The process

🔹 차원 설계 과정 4단계

1. 비즈니스 프로세스를 선택
   - 판매, 재고 관리, 고객 상호작용과 같은 분석하려는 핵심 활동이나 운영을 식별
2. grain 선언(분석의 상세 수준을 정의)
   - 명확할수록 일관성과 확장성을 확보
3. 해당 프로세스의 설명적 속성을 담는 차원을 식별
4. 해당 프로세스와 연결된 정량적 지표나 측정값, 즉 팩트를 식별하는 데 초점

### 차원 모델링에 대한 조언

- 차원 모델링은 사람들이 비즈니스를 관찰하는 방식에 잘 들어맞음
  - 비즈니스 프로세는 측정값(팩트)과 그것을 둘러싼 맥락(차원)
- 다른 접근법보다 전달까지 걸리는 시간이 더 짧을 수 있으므로, 시간이 부족하면 차원 모델링은 좋은 선택
- 모델링이 특정 분석 요구사항에 맞춰 설계되므로, 새로운 요구사항에는 다시 모델링해야할 수 있음
- 차원 모델링은 이미 검증된 기법이므로 모든 데이터를 클라우드 DW에 그냥 넣는 것보다 이 기법을 사용하는게 좋음
- 차원 모델링 외에도 기업은 인먼이나 Data Vault 같은 다른 접근법도 데이터 분석에 사용
  - 데이터를 어떻게 모델링할지는 비즈니스에 따라 결정해야함
- One Big Table의 가치는 그 아래에 신중한 데이터 모델링 레이어가 있을 때 있음
  - 처음부터 모든 데이터를 하나의 테이블에 넣는 것은 데이터의 이해하기 어려움
