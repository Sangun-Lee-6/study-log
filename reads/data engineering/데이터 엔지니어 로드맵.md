# The Data Engineer Roadmap

# 1. 소개

- 데이터를 A에서 B로 이동시키는 것은 간단하지 않음
  - 다음과 같은 어려움이 있음
    - 원시 데이터에서 인사이트를 추출
    - 대규모 데이터를 안정적으로 수집, 변환, 저장 및 제공
  - 게다가 데이터 소비자는 이제 사람만 있는게 아니라 AI 모델까지 있음
- 그리고 데이터 제품의 고유한 특성으로 인해 프로젝트마다 신뢰성, 확장성, 성능 및 사용성에 대한 요구사항이 다르므로 어려움
- 시작하기 전 참고 사항
  - 데이터 엔지니어의 책임에 대해 이해하기(링크)
  - AI 에이전트가 파이프라인을 구축하려면, 아직까진 개발자의 검토가 필요
  - 이 글은 기술적인 능력/도구에만 초첨을 두고 있는데, 이 외에 문제 해결 능력⋅소통 능력⋅사용자 중심 사고방식도 중요한 요소임을 인지

# 3. 데이터 모델링

### 📌 원시 데이터에서 인사이트를 얻기 위한 질문

1. 데이터를 어떻게 변환하는가?
2. 데이터를 효율적으로 제공하고 활용할 수 있도록 어떻게 구성해야하는가?

### 📌 큰 그림을 보지 못하면 생기는 문제

- 초기에는 사례별로 접근할 수 있음
  - 사용자마다 필요한 테이블을 위한 파이프라인을 구축 → 작동은 하지만 사용자가 늘어나고, 데이터 소스가 늘어나고, 데이터 제공 방식이 변경되면 문제가 발생
- 큰 그림을 보지 못하면 문제점이 생김
  - 모호한 용어(ex. 재무 부서의 "고객"과 영업 부서의 "고객"의 차이 등)
  - 사일로화된 데이터
  - 새로운 데이터 소스를 추가하는 데 몇 주 이상 소요
  - 동일한 지표에 대한 결과가 달라짐
  - 느린 쿼리 속도
  - 비즈니스 사용자의 신뢰성 부족

### 📌 데이터 모델링 by 조 라이스

- 데이터 모델링은 데이터를 기반으로 업무와 분석이 안정적으로 가능해지도록 형식을 만드는 것
  > _데이터 모델은 인간과 기계의 행동을 가능하게 하고 안내하며, 의사결정에 정보를 제공하고, 실행을 용이하게 하기 위해 데이터를 구조화하고 표준화하는 구조화된 표현 방식입니다. — 조 라이스_
  - _“데이터를 구조화하고 표준화한다”_
    - 제약 없이 저장하지 않고 “이 값은 고객 ID”, “상태값은 이런 종류만 허용”처럼 형태와 규칙을 정하는 것
  - _“인간과 기계의 행동을 가능하게 하고 안내한다”_
    - 사람도 데이터를 보고 이해하고, 프로프램도 그 데이터를 믿고 처리할 수 있게 해줌
    - 테이블 구조가 명확하면 코드, 분석, 배치 시스템이 올바르게 동작 가능
  - _“실행을 용이하게 한다”_
    - 단순히 보기 좋게 정리하는게 아니라 서비스 운영, 분석, 자동화와 같은 업무가 동작하도록
  - 비유하면 데이터 모델은 회사의 문서 양식과 같음
    - 연차 사용 신청서란 문서 양식이 있다면, 각 항목에 정해진 입력 형식이 있음
    - 이런 형식이 있으면 사람이 검토하기도 쉽고 프로그램이 집계하기도 쉬움

### 📌 데이터 모델링의 장점

- 공통 언어 : 데이터 모델링을 통해 명확한 의사소통이 가능
- 데이터 품질 및 무결성 : 제약 조건 및 관계 모델링으로 데이터 품질을 보장
- 오류 감소

### 📌 시멘틱 레이어를 사용한다고 데이터가 좋아지지 않는다, 데이터 모델링이 우선이다

- 시멘틱 레이어란 원본 데이터를 이해하고 쓰기 쉽게 해주는 일종의 번역기
  - ex. `cust_no` 는 보기 어려우니까 `고객ID` 로 바꿔서 보여줌
  - ex. 사용자가 `신규 고객 수` 같은 지표를 선택하면 SQL을 통해 결과를 보여줌
- 그러나 원본 데이터가 엉망이면 시멘틱 레이어를 사용해도 결과가 엉망
  - ex. 원본 데이터의 상태가 주문 기준이 팀마다 다르고, 매출 계산 방식이 부서마다 다르다면 시멘틱 레이어를 사용해도 잘못된 결과가 나옴
- 이 순서가 중요함 : 비즈니스 프로세스 → 데이터 모델링 → 시멘틱 레이어
  - 비즈니스 프로세스 : 회사가 어떻게 일하는가
    - ex. 주문은 어떻게 발생하는가? 고객은 어떤 기준으로 정의되는가?
  - 데이터 모델링 : 비즈니스 로직을 데이터 구조로 옮기는 작업
    - ex. 주문 fact 테이블, 고객 디멘젼 테이블 등
  - 시멘틱 레이어 : 사용자가 쉽게 쓰도록 이름과 의미 붙이기
    - ex. `gross_sales` → 총매출

### 📌 시멘틱 레이어의 두 단계

1. 선언 단계
   - 무엇을 어떻게 볼지 정의
   - ex. 어떤 테이블을 사용할지, 테이블끼리 어떻게 연결할지, 매출/고객 수
   - 즉, 설정 단계
2. 소비 단계
   - 사용자가 실제로 지표를 선택
   - 그러면 시스템이 선언 단계에서 정해둔 내용을 바탕으로 SQL 생성
   - 즉, 사용 단계

- 소비 단계의 품질은 선언 단계에 달려 있음
  - 사용자가 BI 화면에서 아무리 쉽게 클릭해도 정의된 내용이 잘못되어 있으면 결과도 잘못됨
  - 그래서 데이터 모델링이 중요함
  - 데이터 모델링은 비즈니스가 데이터로 어떻게 표현될지 결정하는 것

### 📌 차원 모델링을 먼저 학습하기

- 직관적이고 실무에서 널리 쓰이는 모델링
- 분석하기 좋은 형태로 데이터를 정리할 수 있음
- 차원 모델링은 Fact와 Dimension으로 나뉨
- Fact : 숫자, 사건, 측정값
  - ex. 주문 건수, 매출액
- Dimension : 숫자를 보는 기준
  - ex. 날짜, 고객, 상품, 지역

### 📌 차원 모델링 실습

- 링크

### 📌 킴볼 모델링

- 비즈니스 프로세스를 fact, dimension 구조로 바꾸는 방식
- 비즈니스 프로세스 단위로 모델을 생성
  - ex. 주문 분석 모델, 결제 분석 모델

# 4. SQL

### 📌 데이터 처리 중심의 이동 : 애플리케이션 코드 → 데이터 플랫폼 내부

- OLAP DB의 발전과 dbt 같은 도구로 SQL 기반 변환 작업을 체계적으로 관리 가능
  - 따라서 많은 변환 작업을 굳이 애플리케이션 코드에서 하지 않고 DB 안에서 SQL로 직접 처리하는게 더 단순하고 효율적이게 됨
- 빅쿼리 같은 플랫폼 안에서 머신러닝도 돌릴 수 있음

### 📌 SQL 가이드

- 데이터 모델링과 함께 SQL을 학습하면 2가지를 동시에 연습할 수 있음
- 그룹화, 집계, 정렬, 다양한 유형의 조인, CTE, 윈도우 함수를 숙지할 것
- SQL 쿼리 내부 동작 방식도 이해할 것 → 쿼리 디버깅 및 최적화를 위함
- 확장 가능하고 읽기 쉽고 실행 가능한 쿼리를 작성해야 함

# 5. 파이썬

### 📌 파이썬 활용

- 반복적인 작업 자동화
- SQL로 표현하기 어려운 복잡한 변환 작업 처리(Python, Pandas, Polars)
- REST API로 여러 시스템에서 데이터를 가져오기
- 데이터 파이프라인 단계 조율(ex. Airflow의 Dag 작성)
- FastAPI 같은 프레임워크로 데이터 애플리케이션 구축

### 📌 파이썬 학습 가이드

- 기본 문법 외에 읽기 쉽고 유지보수 및 확장이 가능한 코드를 작성해야 함
- 동료들이 이해하지 못하거나 확장하는데 어려움이 있다면 훌륭한 코드라도 쓸모 없음
- NumPy, Pandas를 이용한 데이터 처리에 익숙해지고 API 호출부터 연습(링크)
- 체계적인 코드를 작성하는데 집중하기(디자인 패턴, SOLID, 클린 코드)(링크)

# 6. OLAP

### 📌 데이터 파이프라인의 목적지 : OLAP DB

- 데이터 파이프라인의 가장 일반적인 목적지
- 이는 데이터 웨어하우스의 물리적 구현체
- 변환된 데이터는 최종 사용자에게 제공되기 위해 OLAP DB에 저장됨

### 📌 OLTP DB

- 소량의 데이터를 처리하는 데이터 작업을 위해 설계됨
- 단일 데이터를 최대한 빠르게 찾도록 최적화되어 있음
- 데이터는 행 형식으로 구성되고 같은 행에 있는 값들은 함께 저장됨
- ex. PostgreSQL, MySQL

### 📌 OLAP DB

- 분석에 필요한 데이터를 저장⋅처리⋅검색하는 DBMS
- 대규모 데이터를 로드해서 집계 및 조인을 수행하도록 설계됨
- 데이터는 컬럼 형식으로 구성됨
- ex. BigQuery, Snowflake

### 📌 클라우드 DW

- Snowflake, Databricks, BigQuery와 같은 클라우드 DW 무료 평가판에 가입해서 학습
- 데이터를 불러오고 사용자 인터페이스/사용자 경험(UI/UX)에 익숙해지며, 데이터 구성 방식(예: 데이터베이스, 스키마)과 쿼리 모니터링 및 디버깅 방법을 익히기
- 비용과 성능을 최적화하기 위한 최적의 방법을 이해하고 적용(ex. 클러스터링, 파티셔닝)
- 학술 논문이나 공급업체 자료를 통해 내부 작동 방식을 파악하기

### 📌 DuckDB

- 클라우드 DW와 달리 DuckDB는 최소한의 설정만으로 노트북에서 직접 실행되는 OLAP 시스템

### 📌 Open Table Format

- 클라우드 DW는 편리하지만 데이터 관리 방식이 공급업체에 종속되는 경우가 많음
- 따라서 새로운 쿼리 엔진이나 처리 도구를 붙이려고 할 때 제약이 생길 수 있음
- 이러한 문제를 해결하는 방식이 레이크 하우스
- 레이크하우스는 데이터를 객체 스토리지에 저장하면서, DW처럼 테이블 단위로 관리하고 분석 가능
- 이때 핵심이 되는 기술이 Open Table Format
- 대표적으로 Iceberg, Hudi, Delta Lake가 있음
- Open Table Format은 객체 스토리지 위의 데이터를 단순 파일이 아니라 관리 가능한 테이블처럼 다룰 수 있게 해줌
- ex. 스키마 변경, 파티션 관리, 버전 관리 같은 기능을 지원
- 레이크하우스의 장점 2가지

1. 데이터를 기업이 직접 관리하는 스토리지에 저장 가능
2. 같은 데이터를 여러 쿼리 엔진에서 활용할 수 있는 기반

- 처음에는 OLAP 시스템의 기본 개념을 익히고 Iceberg의 구조를 깊이 이해해보기(특히 메타데이터를 어떻게 관리하는지 이해하는 것이 핵심)

# 7. dbt

### 📌 과거 데이터 웨어하우스 시스템과 ETL 방식

- 과거에는 데이터 웨어하우스 시스템이 비쌌고, 확장성도 낮고, 행 기반 DB라서 분석용 작업에 적합하지 않았음
- 이러한 환경에선 ETL 방식이 가장 현실적인 해결책
- 원천 데이터에서 필요한 데이터를 먼저 추출하고, 미리 가공한 뒤, 정제된 일부 데이터만 DW에 적재

### 📌 현대의 데이터 웨어하우스 시스템과 ELT 방식

- 클라우드 DW와 DuckDB 같은 OLAP 등장으로 분석용 데이터 처리 역량이 보편화됨
- 저장 비용이 낮아지고 네트워크 속도도 증가하고 컬럼 기반 저장⋅처리
- 이로 인해 원천 데이터를 먼저 웨어하우스에 그대로 적재한 뒤, 그 안에서 나중에 SQL로 변환이 가능해짐
- 따라서 SQL이 데이터 변환의 중심 수단으로 많이 사용되고 있음

### 📌 SQL 쿼리를 체계적으로 관리하기 위한 도구 : dbt

- 이러한 흐름 속에서 SQL 쿼리를 체계적으로 관리할 방법이 필요해짐
- 이때 사용하는 도구가 dbt
- dbt는 SQL 데이터 변환 작업을 효율적으로 관리할 수 있게 해주는 CLI 도구
- dbt는 실행 엔진이 아니고, DB도 아니라 SQL 기반 데이터 변환 작업을 구조적으로 관리해주는 도구

### 📌 dbt 기본 개념

- dbt의 핵심 개념 : model
- model은 .sql 파일에 저장된 하나의 SQL 쿼리라고 보면 됨
- dbt model의 코드는 순수 SQL만으로 이루어지지 않고 Jinja 템플릿 문법이 함께 사용됨

### 📌 dbt의 Jinja 함수 예시 : source(), ref()

- source() : 데이터 웨어하우스 안에 실제로 존재하는 원본 테이블을 참조할 때 사용
- ref() : 다른 dbt model을 참조할 때 사용
- 이 둘을 활용하면 dbt 안에서 데이터 변환 흐름은 연결 가능
- 예시 : 처음 model은 source()를 통해 원본 테이블을 읽고, 그 다음 model은 ref()를 사용해 앞선 model의 결과를 참조
- 이러면 원본 데이터로부터 최종 결과 테이블까지의 변환 계보(데이터 리니지)를 명확하게 만들 수 있음

### 📌 dbt의 장점

- dbt model은 결국 코드 파일이므로 Git과도 매우 잘 맞음
- 애플리케이션 코드처럼 CI/CD 파이프라인을 구성해 테스트, 빌드, 배포 자동화 가능
- dbt는 데이터 변환 구조와 lineage를 한 눈에 볼 수 있는 문서도 자동으로 생성해줌
