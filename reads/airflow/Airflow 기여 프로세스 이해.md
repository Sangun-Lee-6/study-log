# Airflow 기여 프로세스 이해

### 1. 전체 흐름

```
GitHub Issue 확인/AIP 확인
→ Airflow 저장소 Fork
→ 로컬 개발환경 세팅
→ 브랜치 생성
→ 코드 수정 + 테스트 작성
→ static check / test 실행
→ Pull Request 생성
→ 리뷰 반영
→ maintainer 승인
→ merge
```

---

### 2. 기여 초보자가 특히 이해해야 할 포인트

1. PR을 먼저 만들 수도 있다
   - Airflow는 항상 이슈를 먼저 만들 필요 없음
     - 작은 문서 수정, 테스트 보강, 명확한 버그 수정은 바로 PR 가능
   - 다만 설계 논의가 필요한 변경이면 이슈를 만들고 의견을 받는게 좋음
2. Breeze는 Airflow 전용 개발환경이다
   - Airflow는 프로젝트가 크므로, 로컬 파이썬 환경만으로 맞추기 어려움
   - 따라서 공식 문서는 도커 기반 Breeze 환경을 사용해 CI와 비슷한 환경에서 개발하고 테스트할 수 있음
3. 테스트가 안 통과하면 리뷰가 잘 안 붙는다
   - Airflow는 PR에서 static check와 test가 “1차 품질 기준” 역할
   - 즉, CI가 빨간 상태면 maintainer가 본격 리뷰를 하지 않을 가능성이 높음
   - 먼저 관련 테스트와 static check를 통과시키는 게 중요
4. 리뷰는 “지적”이 아니라 “합의 과정”이다
   - maintainer가 코멘트는 코드 품질, 프로젝트 방향성, 기존 설계와의 일관성을 맞추는 과정
   - 반영할 건 반영하고, 다르게 생각하면 근거를 들어 설명
