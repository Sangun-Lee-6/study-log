## 기여 목적

- 데이터 파이프라인의 재처리, 검증, 실패 방지, 운영 일관성을 개선한 기여
- 이를 통해 데이터 파이프라인이 실패했을 때 어디서 깨졌는지 추적하고, 검증 로직을 고치고, 재발 방지 테스트까지 남길 수 있음을 증명

---

## 기여의 가치

- 장애 상황을 이해하는가? ⇒ Scheduler, Backfill, DagRun, TaskInstance, retry, state transition 관련 기여
- 데이터 재처리를 이해하는가? ⇒ Backfill, dry-run, reprocess behavior, catchup 관련 기여
- 운영 안정성을 높이는가? ⇒ validation, API/CLI consistency, regression test 추가
- 협업이 가능한가? ⇒ 이슈 분석, 재현, 테스트, maintainer 리뷰 대응
- 실무 코드베이스를 읽을 수 있나? ⇒ Airflow 내부 모델/API/CLI/test 흐름을 따라가며 수정

---

## 기여 수준

- Level 2-4 수준에서 기여할 것

| 레벨    | 기여 유형                                        | 가치                              |
| ------- | ------------------------------------------------ | --------------------------------- |
| Level 0 | 오타, 표기, 번역 수정                            | 낮음. 진입 경험 정도              |
| Level 1 | CLI help, UI 문구, docs 일관성 수정              | 낮음~보통. 협업 가능성 증명       |
| Level 2 | API/CLI behavior 불일치 수정 + 테스트            | 높음. 운영 안정성 기여            |
| Level 3 | Backfill, Scheduler, DagRun 검증 로직 버그 수정  | 매우 높음. DE 플랫폼 이해도 증명  |
| Level 4 | AIP 후속 작업, 구조 개선, maintainer급 반복 기여 | 강함. 플랫폼 엔지니어링 역량 증명 |

---

## 어떻게 기여해야 가치가 생기는가

- PR 구조
  ```
  1. 문제 정의
     - 어떤 사용자 시나리오에서 문제가 발생하는가?

  2. 재현 경로
     - CLI/API/UI 중 어디서 재현되는가?
     - expected vs actual이 무엇인가?

  3. 원인 분석
     - 어떤 함수/계층에서 값이 누락되거나 불일치하는가?

  4. 수정
     - 최소 변경으로 behavior를 맞춘다.

  5. Regression test
     - 같은 버그가 다시 생기면 CI에서 실패하게 만든다.

  6. 문서/릴리즈 노트
     - 필요한 경우만 보강한다.
  ```
- 버그를 재현하고, 테스트를 추가하기
  - 같은 도메인에 집중하기
