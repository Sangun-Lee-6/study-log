# 2장 데이터 엔지니어링 수명 주기

- 이 책의 주요 목표 : 데이터 엔지니어링을 특정 데이터 기술의 집합의 관점이 아니라 데이터 수명 주기 관리 관점으로 보도록 만드는 것

# 1. 데이터 엔지니어링 수명 주기란?

🔹 데이터 엔지니어링 수명 주기

- 원시 데이터를 사용자가 유용하게 사용할 수 있도록 전환하는 단계로 구성
- 수명 주기의 5단계 : 데이터 생성⋅저장⋅수집⋅변환⋅서빙
- 저장⋅수집⋅변환 단계의 순서는 반복되거나 바뀔 수 있음

🔹 데이터 수명 주기에서 데이터 저장 단계

- 데이터 저장은 수명 주기 전체에 걸쳐 발생함
- 데이터 엔지니어링 수명 주기의 시작 : 원천 시스템에서 데이터를 가져와 저장하는 것
- 그 다음 사용자에게 데이터를 제공하는 목표로 진행됨

## 1.1 데이터 수명 주기 vs 데이터 엔지니어링 수명 주기

- 데이터 엔지니어링 수명 주기는 데이터 수명 주기의 하위집합

## 1.2 데이터 생성

- 원천 시스템 : 데이터 엔지니어링 수명 주기에서 사용되는 데이터의 원본
- ex. IoT 장치, 메세지 큐, DB
- 데엔은 일반적으로 원천 시스템 자체를 소유하거나 제어하지 않음
- 데엔은 원천 시스템의 작동 방식, 데이터 생성 방식, 데이터의 빈도 및 속도, 생성되는 데이터의 다양성을 실무적으로 이해해야 함
- 데엔은 데이터 파이프라인이 중단될 수 있는 변경 사항에 대해 원천 시스템 소유자와 소통해야 함
- 데엔의 가장 큰 과제 : 데이터가 들어오는 출처가 다양하고 제각각이라서 데엔이 이해해야 할 대상이 매우 많은 것

🔹 원천 시스템 평가 : 주요 엔지니어링 고려 사항

- [ ] 데이터 원천의 본질적인 특징은 무엇인가? 데이터 원천은 애플리케이션인가? IoT 장치의 스웜인가?
- [ ] 원천 시스템에서 데이터는 어떻게 유지되는가? 데이터는 장기간 보존되는가? 아니면 일시적이고 빠르게 삭제되는가?
- [ ] 데이터는 어느 정도의 속도로 생성되는가? 초당 몇 개의 이벤트가 발생할까? 시간당 몇 기가바이트인가?
- [ ] 데이터 엔지니어는 출력 데이터에서 어느 정도의 일관성을 기대할 수 있는가? 출력 데이터에 대해 데이터 품질 검사를 실행할 때, 예상치 못한 출력값(예를 들면 null 값)이나 잘못된 데이터 포맷과 같은 데이터 불일치 사례는 얼마나 자주 발생할까?
- [ ] 에러는 얼마나 자주 발생하는가?
- [ ] 데이터에 중복이 포함되지는 않는가?
- [ ] 일부 데이터값이 동시에 생성되는 다른 메시지보다 훨씬 늦게 도착할 수 있는가?
- [ ] 수집된 데이터의 스키마는 무엇인가? 데이터 엔지니어가 데이터를 완전히 파악하려면 여러 테이블 또는 여러 시스템에 걸쳐 조인을 수행해야 하는가?
- [ ] 스키마가 변경되면(예를 들어 새로운 열이 추가되었을 때) 어떻게 대처하고 다운스트림 이해관계자에게 전달할 수 있는가?
- [ ] 원천 시스템에서 데이터를 얼마나 자주 가져와야 하는가?
- [ ] (고객 계정 정보를 추적하는 데이터베이스 등) 상태가 있는 시스템(stateful system)의 경우, 데이터는 정기적인 스냅숏으로 제공되는가? 아니면 변경 데이터 캡처(CDC)로부터의 갱신 이벤트로 제공되는가? 변경은 어떻게 수행되며, 원천 데이터베이스에서 이러한 변경을 어떻게 추적하는가?
- [ ] 다운스트림 사용을 위한 데이터를 전송하는 데이터 제공업체는 누구(무엇)인가?
- [ ] 데이터 원천에서의 데이터 조회가 성능에 영향을 미치는가?
- [ ] 원천 시스템에 업스트림 데이터 의존 관계가 있는가? 이러한 업스트림 시스템의 특징은 무엇인가?
- [ ] 늦거나 누락된 데이터 확인용으로 데이터 품질 검사가 실시되고 있는가?

🔹 데이터 원천에 대해 데이터 엔지니어가 알아야 할 것

- 각 데이터 원천에는 고유한 데이터 양과 데이터 생성 주기가 있으므로 데엔은 원천에서 데이터가 생성되는 방법을 알아야 함
- 원천 시스템의 한계를 이해해야 함(ex. 원천 DB의 분석 쿼리의 성능 문제)

🔹 원천 데이터마다 생김새가 다르고, 그 생김새가 계속 변하므로 데이터 엔지니어에게 어려움

- 스키마 : 데이터 구조에 대한 설계도
- 스키마리스 방식 : 스키마가 아예 없는게 아니라 DB가 엄격하게 강제하지 않는 것(ex. JSON)
- 고정 스키마 방식 : RDBMS, DB가 미리 정한 구조를 강하게 지키는 방식
- 두 방식 다 시간이 지나면서 바뀌고, 그 데이터를 받는 파이프라인도 영향받으므로 파이프라인과 분석 데이터가 깨지지 않도록 관리해야 함

## 1.3 데이터 저장

🔹 스토리지 솔루션 및 데이터 저장

- 데이터를 저장하기 위한 스토리지 솔루션(storage solution)
- 데이터 저장이 복잡한 이유
  1. 클라우드 데이터 아키텍처는 여러 스토리지 솔루션을 사용함
  2. 많은 데이터 스토리지 솔루션은 복잡한 변환 쿼리를 지원함
  3. 데이터 저장은 다른 데이터 수명 주기와 연관되어 있음(수집, 변환, 제공)
- 데이터 저장 방식은 데이터 엔지니어링 수명 주기의 모든 단계에 여러 영향을 미침

🔹 스토리지 시스템 평가 : 주요 엔지니어링 고려 사항

- [ ] 이 스토리지 솔루션은 아키텍처에서 요구하는 쓰기 및 읽기 속도와 잘 맞는가?
- [ ] 스토리지가 다운스트림 프로세스의 병목 현상을 초래하지는 않는가?
- [ ] 이 스토리지 기술이 작동하는 방식을 인지하고 있는가? 스토리지 시스템을 최적으로 활용하는가? 아니면 부자연스러운 행동을 하는가? 예를 들어 객체 스토리지 시스템에 높은 비율의 임의 접근(random access) 갱신을 적용하고 있지는 않은가?(이는 성능 오버헤드가 큰 안티패턴)
- [ ] 이 스토리지 시스템은 향후 예상되는 확장을 처리할 수 있는가? 사용 가능한 총 스토리지, 읽기 작업 속도, 쓰기 볼륨 등 스토리지 시스템의 모든 용량 제한을 고려해야 함
- [ ] 다운스트림 사용자와 프로세스가 필요한 서비스 수준 협약(SLA)에 따라 데이터를 취득할 수 있는가?
- [ ] 스키마 진화, 데이터 흐름, 데이터 계보 등에 대한 메타데이터를 캡처하고 있는가? (메타데이터는 데이터 활용성에 큰 영향을 미침)
- [ ] 순수 스토리지 솔루션(객체 스토리지)인가? 아니면 복잡한 쿼리 패턴(예: 클라우드 데이터 웨어하우스)을 지원하는가?
- [ ] 스토리지 시스템이 스키마에 구애받지는 않는가(객체 스토리지)? 유연한 스키마(카산드라)인가? 강제 적용된 스키마(클라우드 데이터 웨어하우스)인가?
- [ ] 데이터 거버넌스를 위해 마스터 데이터, 골든 레코드 데이터 품질 및 데이터 계보를 어떻게 추적하고 있는가?
- [ ] 법령 준수 및 데이터 주권에 어떻게 대처하고 있는가? 예를 들어 특정 지리적 위치에는 데이터를 저장하고 다른 위치에는 저장하지 않을 수 있는가?

🔹 데이터 접근 빈도 이해

- 데이터 접근 방식은 데이터에 따라 달라짐 → 데이터 접근 빈도에 따라 데이터 온도가 결정됨
- 자주 접근되는 데이터(핫 데이터)는 빠른 검색용으로 저장되어야 함
- 거의 쿼리되지 않는 데이터(콜드 데이터)는 규정 준수 목적으로 보관되거ㅏ 다른 시스템에 장애가 발생했을 때 보관되는 경우가 많음

🔹 스토리지 시스템 선택

- 주요 고려 사항 : 데이터의 사용 사례, 데이터 볼륨, 수집 빈도, 형식 및 크기
- (위에 있는 스토리지 시스템 평가 항목에 대부분 기술되어 있음)
- 모든 스토리지 기술에는 트레이드 오프가 있음

## 1.4 데이터 수집

🔹 데이터 수집 단계의 중요성

- 데이터를 수집하기 전에 데이터 원천, 원천 시스템의 특징 및 데이터 저장 방법을 이해해야 함
- 데이터 수집 단계가 데이터 엔지니어링 수명 주기 중 가장 큰 병목 현상을 나타냄(직접 관리할 수 없으므로)
- 신뢰할 수 없는 원천 시스템은 수명 주기 전반에 걸처 파급 효과가 생길 수 있음
- 수집 단계에서 이슈가 발생하면 데이터 흐름이 멈추거나 저장, 처리, 서비스에 불충분한 데이터가 제공됨

🔹 수집 단계에서의 주요 엔지니어링 고려 사항

- [ ] 수집 중인 데이터의 사용 사례는 무엇인가? 같은 데이터셋의 여러 버전을 생성하는 대신 이 데이터를 재사용할 수 있는가?
- [ ] 시스템이 이 데이터를 안정적으로 생성하고 수집하고 있는가? 필요할 때 해당 데이터를 사용할 수 있는가?
- [ ] 수집 후 데이터 목적지는 어디인가?
- [ ] 데이터에 얼마나 자주 접근해야 하는가?
- [ ] 데이터는 보통 어느 정도의 용량으로 도착하는가?
- [ ] 데이터 형식은 무엇인가? 다운스트림 스토리지 및 변환 시스템에서 이 형식을 처리할 수 있는가?
- [ ] 원천 데이터는 다운스트림에서 즉시 사용할 수 있는 양호한 상태인가? 그렇다면 얼마나 오래 사용할 수 있으며, 사용할 수 없게 되는 요인은 무엇인가?
- [ ] 데이터가 스트리밍 소스에서 전송된 경우, 목적지에 도달하기 전에 데이터를 변환해야 하는가? 데이터가 스트림 자체 내에서 변환되는 형태의 변환이 적절할까?

🔹 배치 vs 스트리밍

- 데이터는 원천에서 지속적으로 생성되고 갱신되므로 대부분 데이터는 본질적으로 스트리밍
- 배치 수집은 이 스트림을 큰 청크로 처리하는 방법
- 스트리밍 수집을 사용하면 다운스트림 시스템에 데이터를 실시간으로 제공 가능
- 배치 수집은 미리 설정된 시간 또는 크기에 도달하면 수집 → 다운스트림에선 항상 일정 시간만큼 늦게 데이터를 받게 됨

🔹 배치와 스트림 수집의 주요 고려 사항

- [ ] 데이터를 실시간으로 수집하면 다운스트림 스토리지 시스템이 데이터 흐름 속도를 처리할 수 있는가?
- [ ] 밀리초 단위의 실시간 데이터 수집이 필요할까? 아니면 매분마다 데이터를 축적하고 수집하는 마이크로 배치 접근 방식이 효과가 있을까?
- [ ] 스트리밍 수집의 사용 사례로는 무엇이 있을까? 스트리밍을 구현하면 구체적으로 어떤 이점을 얻을 수 있을까? 데이터를 실시간으로 가져올 수 있다면, 배치 방식에 비해 개선될 수 있는 데이터에 대해 어떤 조치를 취할 수 있을까?
- [ ] 스트리밍 우선 접근 방식은 단순 배치 방식보다 시간, 비용, 유지 보수, 다운타임 및 기회비용 측면에서 더 많은 비용을 소비할까?
- [ ] 인프라에 장애가 발생했을 때 스트리밍 파이프라인과 시스템이 안정적이고 다중화되어 있는가?
- [ ] 사용 사례에 가장 적합한 도구는 무엇인가? 관리형 서비스(Amazon Kinesis, 구글 클라우드 Pub/Sub, Google Cloud Dataflow)를 사용해야 하는가? 아니면 카프카, 플링크, 스파크 등의 인스턴스를 구축해야 할까? 후자를 선택한다면 누가 관리의 역할을 맡을 것인가? 비용과 트레이드오프는 무엇일까?
- [ ] ML 모델을 배포했을 때 온라인 예측 및 지속적인 훈련으로 얻을 수 있는 이점은 무엇일까?
- [ ] 실제 운영 인스턴스에서 데이터를 가져오는가? 그렇다면 이 원천 시스템에 대한 수집 프로세스의 영향도는 얼마나 될까?

🔹 스트리밍 방식의 트레이드오프

- 스트리밍 방식은 간단하지 않으며 기본적으로 추가 비용과 복잡성이 발생하므로 트레이드오프를 정당화하는 비즈니스를 파악한뒤 적용
- 일반적인 사용 사례에서(모델 훈련, 주간 리포트) 배치 방식으로도 충분함

🔹 푸시 vs 풀

- 푸시(push): 원천 시스템이 타깃이나 중간 저장소로 데이터를 직접 보냄
- 풀(pull): 수집 시스템이 원천 시스템에 가서 데이터를 가져옴
- 하나의 파이프라인 안에서도 구간에 따라 푸시와 풀이 섞여 사용될 수 있으므로 “이 시스템은 무조건 푸시/풀”로 단순하게 나누기보다, 각 단계에서 누가 먼저 데이터를 보내거나 가져오는지로 보는 것이 정확함
- 배치 ETL은 대표적인 풀 방식(수집 시스템이 정해진 주기에 원천 DB를 조회해서 데이터를 가져옴)
- CDC는 푸시와 풀이 모두 가능(변경 발생 시 메시지나 로그를 넘겨주는 방식은 푸시 / 마지막 수집 시점 이후 변경분을 조회하는 타임스탬프 기반 CDC는 풀)
- 로그 기반 CDC의 장점 : 수집 시스템이 DB 테이블을 반복 조회하지 않고, DB가 남기는 변경 로그를 읽으므로 원천 DB에 주는 부하가 적고, 운영 DB에 더 안전한 경우가 많음
- 이벤트 스트리밍은 DB를 우회하는 방식 : 데이터가 먼저 DB에 저장된 뒤 수집되는 것이 아니라, 발생 즉시 이벤트 스트리밍 플랫폼으로 들어감, 이는 실시간 이벤트 처리처럼 현재 상태보다 이벤트 자체가 중요한 경우에 적합
- 배치는 보통 풀 중심, 실시간 이벤트/로그 기반 처리는 푸시
- 어떤 방식이 더 좋은지가 아니라, 원천 시스템 부하·실시간성·데이터 성격에 따라 적합한 방식을 고르는 것이 중요

## 1.5 데이터 변환

🔹 데이터 변환 단계

- 데이터 수집, 저장 뒤 작업하는 단계
- 데이터를 원래 형태에서 다운스트림 사용 사례에 유용한 형태로 변경
- 데이터 수집 직후 기본 변환으로 데이터를 올바른 유형으로 매핑, 표준 형식 지정, 잘못된 유형은 제거
- 변환 단계 이후에 데이터 스키마를 변환하고 정규화 적용 가능

🔹 변환 단계에서의 주요 고려 사항

- [ ] 변환에 드는 비용과 투자수익률(ROI)은 얼마인가? 관련된 비즈니스 가치는 무엇인가?
- [ ] 변환은 가능한 한 단순하고 독립적인가?
- [ ] 변환이 지원하는 비즈니스 규칙은 무엇인가?

🔹 실제 데이터 엔지니어링 수명 주기에서 변환 단계는 훨씬 복잡할 수 있음

- 데이터는 배치 변환하거나 스트리밍 중 변환 가능
- 일반적으로 데이터는 원천 시스템에서 변환되거나 수집 중에 변환됨

🔹 데이터 변환의 여러 원인

- 주요 원인 : 비즈니스 로직
- 데이터는 비즈니스 로직을 재사용 가능한 요소로 변환함
- ex. `판매`라는 비즈니스는 특정 상품을 개당 30달러에 구입했다는 것
- 데이터 모델링은 비즈니스 프로세스를 명확하고 최신화된 상태로 파악할 때 매우 중요
- 다른 원인 중 ML의 데이터 특성화 : ML 모델 훈련에 유용한 데이터 특성을 추출하고 강화하는 것
