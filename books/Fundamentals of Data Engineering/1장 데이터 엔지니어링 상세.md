# 1장 데이터 엔지니어링 상세

# 1. 데이터 엔지니어링이란?

## 1.1 데이터 엔지니어링 정의

- 데이터 엔지니어링은 원시 데이터를 가져와서 사용 사례(ex. 분석, 머신러닝)를 지원하는, 고품질의 일관된 정보를 생성하는 시스템과 프로세스의 개발⋅구현⋅유지 관리
- 데이터 엔지니어는 원천 시스템에서 데이터를 가져와서 사용 사례(ex. 분석, 머신러닝)에 데이터를 제공하는 데이터 엔지니어링 수명 주기를 관리

---

## 1.2 데이터 엔지니어링 수명 주기

![image.png](./img/image1-1.png)

🔹 데이터 엔지니어링 수명 주기의 의미

- 데이터 엔지니어링 수명 주기를 통해 데이터 엔지니어들이 역할을 파악할 수 있음
- 데이터 엔지니어링 수명 주기는 기술이 아니라 데이터와 데이터가 제공해야 하는 최종 목표

🔹 데이터 엔지니어링 수명 주기의 단계

- 데이터 생성
- 데이터 저장
- 데이터 수집
- 데이터 변환
- 데이터 서빙

🔹 드러나지 않는 요소

- 데이터 엔지니어링 수명 주기에 걸쳐 드러나지 않는 요소라는 개념도 포함됨
- 보안, 데이터 관리, 데이터옵스, 데이터 아키텍처, 오케스트레이션, SW 엔지니어링

---

## 1.3 데이터 엔지니어의 진화

- 현재와 미래의 데이터 엔지니어링을 이해하기 위해 해당 분야가 어떻게 발전했는지 파악해야함
- 공통된 주제는 끊임없이 다시 나타나며, 오래된 것들은 다시 새로워짐

🔹 1980년부터 2000년까지: 데이터 웨어하우징에서 웹으로

- 데이터 엔지니어링의 뿌리는 데이터 웨어하우스, BI, ETL
- 이 시기에 SQL, 관계형 데이터베이스, 데이터 모델링이 핵심 기반으로 자리 잡음
- 킴벌·인먼의 모델링 방식은 지금도 중요한 기본기
- 1990년대 웹의 확산으로 데이터가 급격히 늘어나며, 기존 백엔드 시스템이 규모 한계에 부딪히기 시작함

🔹 2000년대 초: 현대 데이터 엔지니어링의 탄생

- 데이터 폭증으로 기존 단일 DB/웨어하우스 방식만으로는 버티기 어려웠음
- 그래서 분산 저장·분산 처리가 핵심 기술로 떠오름
- 구글의 GFS, MapReduce, 야후의 Hadoop, 아마존의 S3, EC2가 현대 데이터 엔지니어링의 기반을 만들었음
- 이 시기부터 데이터 엔지니어는 대용량 데이터를 처리하는 시스템 엔지니어 성격이 강해짐

🔹 2000년대와 2010년대: 빅데이터 엔지니어링

- 하둡, 스파크, 카산드라, 프레스토 등 다양한 빅데이터 도구가 빠르게 확산됨
- 배치 처리 중심에서 점차 이벤트 스트리밍·실시간 처리로 확장
- 하지만 현업에서는 작은 문제에도 빅데이터 기술을 과하게 쓰는 경우가 많았음
- 결국 중요한 건 최신 도구 자체보다 문제 크기에 맞는 기술 선택이라는 점이 드러났음
- 빅데이터 엔지니어의 많은 시간이 가치 전달보다 복잡한 운영과 관리에 쓰이기도 함

🔹 2020년대: 데이터 수명 주기를 위한 엔지니어링

- 최근 흐름은 복잡한 인프라를 직접 운영하기보다 관리형 서비스, 모듈형 도구, 추상화된 스택을 조합하는 방향
- 데이터 엔지니어는 이제 단순 파이프라인 개발자가 아니라 데이터 수명주기 전체를 관리하는 역할로 확장
- 중요 영역도 보안, 데이터 품질, 거버넌스, 오케스트레이션, DataOps, 아키텍처로 넓어졌음
- 이제 핵심은 “큰 데이터를 다룬다”보다 신뢰할 수 있고, 잘 관리되며, 쉽게 활용 가능한 데이터를 만드는 것
- 개인정보보호, 익명화, 규정 준수까지 데이터 엔지니어가 고려해야 하는 시대

🔹 주니어 데이터 엔지니어 관점에서 기억할 점

- SQL과 데이터 모델링은 여전히 가장 중요한 기본기
- 기술 유행보다 왜 이 기술이 등장했는지를 이해하는 게 더 중요함
- 큰 시스템을 만들수록 결국 중요한 건 확장성보다 운영 단순성, 데이터 품질, 신뢰성
- 데이터 엔지니어의 역할의 확장 : 데이터 처리 ⇒ 데이터 수명 주기 관리

---

## 1.4 데이터 엔지니어링과 데이터 과학

🔹 데이터 엔지니어링은 데이터 과학의 업스트림에 위치함

- 데이터 엔지니어는 데이터 과학자가 사용할 입력값을 제공하고, 데이터 과학자는 이를 사용
- 데이터 과학자는 실제 작업 시간의 70%는 데이터 수집⋅데이터 정리⋅데이터 처리에 사용하고, ML 모델 구축 및 튜닝 작업에는 시간을 많이 못 씀
- 데이터 엔지니어가 위 작업을 처리하면 데이터 과학자가 모델 구축 및 튜닝에 집중할 수 있음

# 2. 데이터 엔지니어링 기술과 활동

🔹 데이터 엔지니어의 책임은 파이프라인 너머까지 이어짐

- 데이터 엔지니어링의 드러나지 않는 요소 : 보안, 데이터 관리, 데이터옵스, 데이터 아키텍처, SW 엔지니어링
- 이러한 요소를 사용하려면 데이터 도구들을 평가하는 방법과 이 도구들이 어떻게 조합되는지 이해해야 함
- 그리고 데이터 엔지니어링 수명 주기의 전후 과정도 이해해야 함(원천 시스템에서 어떻게 데이터가 생성되는지, 데이터가 어떻게 소비되고 가치를 만들어내는지)
- 마지막으로 데이터 엔지니어는 복잡한 가변적 요소를 처리하고 비용, 민첩성, 확장성, 단순성, 재사용성, 상호 운용성의 축에 따라 지속해서 최적화를 수행해야 함

🔹 최근의 데이터 엔지니어링 상황

- 예전에는 도구들의 저수준 세부사항까지 이해했음
- 오늘날의 데이터 도구 환경은 관리 및 구현이 간편해짐
- 따라서 데이터 엔지니어는 비즈니스에 가치를 제공하는 것과 균형을 맞춰서 작업을 진행함

🔹 데이터 엔지니어가 하지 않는 일

- ML 모델 구축, 보고서/대시보드 생성, 데이터 분석, KPI 구축, SW 애플리케이션 개발
- 다만 데이터 엔지니어가 일을 잘 수행하려면 위 영역에 대한 적절한 이해도 필요함

---

## 2.1 데이터 성숙도와 데이터 엔지니어

🔹 데이터 성숙도(data maturity) : 데이터가 회사의 경쟁력이 되고 있는 정도

- 데이터가 경쟁 우위로 활용되는 방식
- 데이터 성숙도는 데이터를 실제로 얼마나 잘 쓰고 있는가를 의미함
- 회사가 얼마나 오래됐는지, 매출이 얼마나 큰지를 의미하지 않음
- 스타트업이 대기업보다 데이터 성숙도가 높을 수 있음

🔹 데이터 성숙도와 데이터 엔지니어링의 복잡도

- 회사가 데이터를 쓰는 수준이 높으면, 데엔이 만들어야 할 시스템도 더 복잡해짐
- 데이터를 핵심 자산으로 쓰는 조직은 더 정교한 데이터 엔지니어링이 필요

🔹 데이터 성숙도의 핵심 3가지

- 활용률 : 데이터를 실제 의사결정과 운영에 자주 쓰는가
- 기능 : 데이터를 수집⋅저장⋅분석할 수 있는 체계와 역량이 있는가
- 통합 : 데이터가 팀과 시스템에 끊기지 않고 연결되어 있는가

🔹 데이터 성숙도 모델

- 책에서 제안하는 3단계의 데이터 성숙도 모델
- 데이터로 시작하기, 데이터로 확장하기, 데이터로 선도하기
- 각 단계마다 데엔이 수행하는 작업이 있음

---

🔹 1단계 : 데이터로 시작하기

- 데이터 활용이 아직 초기 단계이며, 목표와 요청이 불명확하고 일회성인 경우가 많음
- 데이터 아키텍처·인프라·활용률이 모두 초기 수준이고, 데이터 팀 규모도 작음
- 이 단계의 데이터 엔지니어는 제너럴리스트로 일하며, 빠르게 가치와 성과를 만들어야 함
- 가장 중요한 일은 견고한 데이터 기반을 먼저 만드는 것임
- 기반이 약한 상태에서 ML부터 먼저 하는 것은 보통 비추천임

🔹 1단계에서 데이터 엔지니어가 집중할 것

- 경영진을 포함한 주요 관계자로부터 지원 확보
- 적절한 데이터 아키텍처 정의
- 회사가 데이터로 해보려는 과제(데이터 이니셔티브)에 필요한 데이터를 먼저 찾고, 그 데이터가 실제로 쓸 만한 상태인지 점검
- 앞으로 분석가/데이터 과학자가 활용할 수 있는 신뢰 가능한 데이터 기반 구축

🔹 1단계에서 데이터 엔지니어가 주의할 것

- 초기에 성과를 내야 조직 내에서 데이터의 중요성이 확립됨
- 그러나 조기 성과는 기술 부채를 야기할 수 있음을 주의
- 데이터 팀 외부의 현업 및 이해관계자와 계속 소통해야함
- 불필요하게 복잡한 시스템보다 단순하고 바로 쓸 수 있는 솔루션을 우선
- 맞춤형 개발은 경쟁 우위를 만드는 경우에만 수행

---

🔹 2단계 : 데이터로 확장하기

- 회사는 임시 데이터 요청 업무가 아닌 공식적인 데이터 요청 관행이 있음
- 다음 과제가 확장성 있는 데이터 아키텍처를 구축하고, 데이터 중심 미래를 계획하는 것

🔹 2단계에서 데이터 엔지니어의 목표

- 공식적인 데이터 관행 수립
- 확장성 있고 견고한 데이터 아키텍처 구축
- 데브옵스 및 데이터옵스 관행 채택
- ML을 지원하는 시스템 구축
- 차별화되지 않은 과중한 업무를 피하고, 경쟁 우위를 확보할 때만 커스터마이징

🔹 2단계에서 데이터 엔지니어가 주의할 것

- 신규 기술은 대부분 도입에 많은 시간과 에너지가 필요하므로, 모든 기술적 의사 결정(ex. 최첨단 기술 도입)은 고객에게 제공되는 가치에 따라 결정
- 초기 병목은 기술보다 데이터 팀의 인력과 처리량이므로 배포⋅운영이 쉬운 솔루션을 우선 선택할 것
- 멋진 기술을 보여주기보다 지속 가능하게 운영 가능한 구조를 만드는 게 더 중요함
- 중요한 것은 멋진 기술보다 조직이 실제로 데이터를 쓰게 만드는 것
- 다른 팀과 데이터 활용 방법과 가치를 계속 설명하고 교육할 것
- 조직이 다음 데이터 성숙 단계로 올라가도록 도울 것

---

🔹 3단계: 데이터로 선도하기

- 데이터 중심 기업
- 자동화된 파이프라인과 시스템으로 분석과 ML 수행 가능한 상태
- 새로운 데이터 소스를 매끄럽게 배포하고 실질적인 가치 창출 가능

🔹 3단계에서 데이터 엔지니어가 수행할 것

- 새로운 데이터의 매끄러운 배포와 사용을 위한 자동화 구축
- 회사의 비즈니스 경쟁력을 만드는 부분은 직접 사용자 도구와 시스템 구축
- 데이터 관리(데이터 거버넌스와 품질 포함)
- 데이터 카탈로그, 데이터 리니지 도구, 메타데이터 관리 시스템을 포함해 데이터를 조직 전체에 전파하는 도구 도입
- (데이터 거버넌스) : 데이터를 누가, 어떤 기준으로, 안전하고 일관되게 관리할지 정하는 체계
- (데이터 카탈로그) : 조직 안의 데이터가 어디에 있고, 무엇을 뜻하는지 찾기 쉽게 정리한 안내서
- (데이터 리니지) : 데이터가 어디서 왔고, 어떤 과정을 거쳐 현재 형태가 되었는지 보여주는 흐름 정보

🔹 3단계에서 데이터 엔지니어가 주의할 것

- 현재 상태에 안주하지 않고 유지 보수와 개선에 집중 → 이전 단계로 후퇴할 위험이 있음
- 여러 기술을 도입하는게 큰 위험 요소이므로 비즈니스적 가치를 제공하는 경우에만 직접 구축한 기술을 도입

---

## 2.2 데이터 엔지니어의 배경과 기술

🔹 데이터 엔지니어에게 필수적인 지식

- 데이터와 기술을 모두 이해해야 함
- 데이터 관리의 모범 사례와 데이터 도구의 옵션, 상호 작용을 알아야 함
- 이를 위해 SW 엔지니어링, 데이터 옵스 및 데이터 아키텍처를 이해해야 함

## 2.3 비즈니스 책임

🔹 비기술자 및 기술자와의 커뮤니케이션 방법 파악

- 커뮤니케이션이 핵심: 기술보다도 사람들과 신뢰를 만들고, 조직의 보고 체계·소통 방식·사일로를 이해하는 것이 중요함

🔹 비즈니스 요건과 제품 요건을 살펴보고 수집하는 방법 이해

- 요건 파악이 먼저: 무엇을 왜 만들어야 하는지, 이해관계자와 같은 기준으로 맞춰야 함

🔹 애자일, 데브옵스, 데이터옵스의 문화적 기반 이해

- 애자일·데브옵스·데이터옵스는 문화: 도구만 도입한다고 해결되지 않고, 조직 차원의 합의와 협업 방식이 중요함

🔹 비용 관리

- 비용 관리도 중요한 역량: 가치, 속도, 총비용을 함께 보고 지속적으로 모니터링해야 함

🔹 지속적 학습

- 지속적 학습이 필수: 기본기를 갖춘 상태에서, 새 기술 중 무엇이 실용적이고 무엇이 유행인지 구분할 줄 알아야 함

🔹 항상 큰 그림을 그리는게 중요

- 결국 큰 그림이 중요함: 좋은 데이터 엔지니어는 기술만 잘하는 사람이 아니라, 비즈니스 가치와 조직 전체 흐름까지 이해하는 사람임

## 2.4 기술 책임

🔹 데이터 엔지니어의 기술 책임 : 데이터 엔지니어링 수명 주기를 지원

- 데이터 엔지니어링 수명 주기를 지원하는 데이터 아키텍처 구축하기
- 데이터 엔지니어링의 수명 주기
  - 데이터 생성
  - 데이터 저장
  - 데이터 수집
  - 데이터 변환
  - 데이터 서빙
- 데이터 엔지니어링 수명 주기(드러나지 않는 요소)
  - 보안
  - 데이터 관리
  - 데이터옵스
  - 데이터 아키텍처
  - 오케스트레이션
  - SW 엔지니어링

🔹 데이터 엔지니어로서 필요한 데이터와 기술

- 데이터 엔지니어도 코딩할 줄 알아야함
- 사용 제품 수준의 SW 엔지니어링을 갖춰야함
- 이제는 SaaS 제품으로 저수준 프로그래밍 작업 대신 추상화나 오케스트레이션에서 파이프라인 코드를 작성하는 작업에 더 집중
- 특정 기술 요구 사항이 발생했을 때 코드베이스 아키텍처를 볼 수 있어야함\

🔹 데이터 엔지니어가 알아야 할 프로그래밍 언어

- 주요 언어 : SQL, 파이썬, JVM 언어(Java, 스칼라), Bash
- SQL : DB, 데이터 레이크의 가장 일반적인 언어
- 파이썬 : 데이터 엔지니어링⋅데이터 과학에서 사용되는 언어, 각종 데이터 도구의 언어
- JVM 언어(Java, 스칼라) : 아파치 오픈 소스에 사용됨(Spark, Hive 등), 파이썬보다 성능 우수, 파이썬보다 저수준 특성에 접근 가능
- Bash : 리눅스 CLI 언어, 스크립트 작성⋅OS 명령
- 보조 언어 : R, JS, Go, Rusy, C/C++, C# → 회사에서 사용되는 주요 언어이거나 도메인 고유 데이터 도구와 함께 사용 가능

🔹 빠르게 변하는 분야에서 뒤처지지 않는 법

- 변화하지 않는 것을 이해할 수 있도록 기본에 충실하되, 어느 방향으로 발전할지 지속적인 관심을 갖기
- 새로운 기술이 데이터 엔지니어링 수명 주기에 어떻게 도움이 되는지 이해하기

🔹 SQL의 불합리한 유효성

- SQL은 지금도 데이터 변환과 분석의 핵심 언어
- SQL만 잘한다고 충분하지 않음, Spark, dbt와 같은 도구와 함께 SQL을 어떻게 연결해 쓰는지까지 알아야 파이프라인을 만들 수 있음
- 최신 데이터 환경에서는 JSON 파싱, 중첩 데이터 처리, 모델 간 SQL 관리 능력이 중요
- SQL이 적합하지 않은 상황도 구분할 수 있어야함(ex. 텍스트 전처리와 같이 절차 로직이 많다면 SQL보단 파이썬)

## 2.5 A에서 B로 이어지는 데이터 엔지니어링 역할의 연속성\

- 데이터 엔지니어는 모두 같은 일을 하는 사람이 아님
- 회사의 데이터 성숙도와 문제 성격에 따라 상용 도구를 잘 조합하는 역할과 직접 시스템을 만드는 역할로 나뉠 수 있음
- A(Abstraction)형 데이터 엔지니어 : 관리형 서비스와 검증된 도구를 활용해 빠르고 단순하게 데이터 기반을 생성, 초기에 필요한 파이프라인, 적재, 운영 체계를 안정적으로 구축하는 역할
- B(Build)형 데이터 엔지니어 : 회사의 경쟁력이 되는 데이터 도구나 시스템을 직접 구축하는 역할, 일반 솔루션으로 해결되지 않는 문제, 맞춤형 플랫폼, 핵심 데이터 시스템이 필요할 때 중요
- 실무에선 A형 역량이 먼저 필요하고 이후 회사가 성장하거나 요구사항이 복잡해지면 B형 역량의 비중이 커짐
- 따라서 주니어는 먼저 표준 도구를 잘 써서 기반을 만드는 능력을 먼저 익히고, 이후 필요에 따라 직접 구축하는 방향으로 확장

# 3. 조직 내 데이터 엔지니어

- 데이터 엔지니어는 혼자 일하는 역할이 아니라 기술자, 비기술자와 계속 협업하는 역할
- 따라서 파이프라인만 만드는 것이 아니라, 누구를 위한 데이터인지⋅어떤 요구사항을 만족해야 하는지를 이해해야함

## 3.1 내부 vs 외부 대면 데이터 엔지니어

- 데이터 엔지니어의 업무 대상은 크게 외부 사용자/내부 조직으로 나눌 수 있음
- 어떤 대상이냐에 따라 시스템 설계와 운영 방식이 달라짐
- 외부 대면 데이터 엔지니어링 : 외부에서 발생하는 이벤트(앱 사용자, IoT 기기 등)와 트랜잭션 데이터를 다루고 다음과 같은 운영 요구사항이 중요(대규모 동시성⋅엄격한 쿼리⋅보안⋅멀티테넌시)
- 내부 대면 데이터 엔지니어링 : 사내 의사결정과 분석을 지원하는 데이터 시스템(BI⋅리포트⋅DW⋅ML용 데이터 파이프라인)
- 실무에서는 내부, 외부가 완전히 분리되지 않고 서로 연결되는 경우가 많으므로, 같은 데이터라도 사용자에 따라 성능⋅보안⋅접근방식⋅요구사항이 달라짐

---

## 3.2 데이터 엔지니어와 기타 기술 역할

- 데이터 엔지니어는 혼자 일하지 않고 데이터 생산자와 데이터 소비자 사이를 연결하는 허브
- 따라서 데이터를 만드는 쪽과 데이터를 쓰는 쪽 모두를 이해해야 함
- 실무에서는 파이프라인 구현만 중요한게 아니라 어디서 데이터가 만들어지고 누가 그것을 어떻게 쓰는지 파악하는 능력이 중요

### 업스트림 이해관계자

🔹 데이터 아키텍트

- 조직의 데이터 구조와 운영 원칙을 설계하는 역할
- 데엔은 이 아키텍처 방향 안에서 시스템과 파이프라인을 구현하는 경우가 많음
- 따라서 데엔도 단순 구현이 아니라 데이터 아키텍처와 거버넌스의 기본 개념은 이해하고 있어야함

🔹 소프트웨어 엔지니어

- SW 엔지니어는 데엔이 다루는 원천 데이터의 주요 생산자
- 애플리케이션 DB, 이벤트, 로그가 어떻게 생성되는지 알아야 수집/적재/분석 가능

🔹 데브옵스 엔지니어와 사이트 신뢰성 엔지니어

- 이들은 운영 과정에서 모니터링⋅로그⋅메트릭 데이터를 만듦
- 이 데이터는 데이터 엔지니어링의 입력이 되기도 하고, 반대로 데이터 시스템 운영에서 함께 협업해야 하는 대상이 되기도 함

### 다운스트림 이해관계자

- 데이터 엔지니어링의 목적은 결국 데이터를 쓰는 사람들이 바로 활용할 수 있게 만드는 것

🔹 데이터 과학자

- 데이터 과학자는 예측, 추천 모델을 만들지만, 실제로는 데이터 수집⋅정제⋅준비에 많은 시간을 빼앗기기 쉬움
- 따라서 데엔의 역할은 이 과정을 자동화⋅재사용 가능하게 만들어 데이터 과학자가 모델링 자체에 집중할 수 있게 만드는 것

🔹 데이터 분석가

- 데이터 분석가는 주로 현재와 과거의 비즈니스 현황을 이해하는 역할
- SQL, BI 도구를 사용하며 데이터 정의와 품질 문제에 대해 이해하고 있음
- 데엔은 분석가와 협업해서 분석에 필요한 데이터 파이프라인을 생성
- 특히 데이터 분석가는 어떤 데이터가 실제로 필요한지 잘 알고 있으므로, 데이터 품질과 모델링을 맞추는데 중요

🔹 머신러닝 엔지니어와 인공지능 연구원

- ML 엔지니어는 모델 학습⋅배포⋅운영 인프라를 다루며, 데엔과 역할이 일부 겹침
- 실무에서는 데이터 파이프라인과 ML 시스템 운영이 연결되므로 협업이 중요
- 데엔은 ML을 위한 학습⋅추론에 필요한 데이터를 안정적으로 공급하는 역할을 수행

## 3.3 데이터 엔지니어와 비즈니스 리더십

- 데이터 엔지니어는 기술 구현만 하지 않고, 비즈니스와 데이터를 연결

🔹 최고 경영진의 데이터

- 이제 데이터와 분석은 IT만의 관심사가 아니라 최고 경영진이 챙기는 경영 자산
- 그래서 데엔도 단순 개발자가 아니라 데이터로 무엇이 가능한지 설명할 수 있는 역할을 수행

🔹 최고 경영자(CEO)

- CEO는 기술 상세보다 데이터가 비즈니스에 어떤 가치를 주느지에 관심이 있음
- 데엔은 조직이 어떤 데이터를 언제 활용할 수 있는지 보여주고, 클라우드 전환⋅신규 데이터 시스템 도입 같은 변화를 진행

🔹 최고 정보 책임자(CIO)

- CIO는 IT와 비즈니스 프로세스를 함께 책임지며, 전사 차원의 기술 방향과 주요 시스템 도입을 결정
- 데엔은 CIO 조직과 연결되어 데이터 문화⋅클라우드 마이그레이션⋅핵심 시스템 연계에 관여 가능

🔹 최고 데이터 책임자(CDO)

- CDO는 회사의 데이터 자산과 데이터 전략을 책임지는 역할
- 데이터 제품⋅거버넌스⋅개인정보보호⋅마스터 데이터 관리처럼 데이터 자체를 비즈니스 자산으로 다루는 관점

🔹 최고 분석 책임자(CAO)

- CAO는 주로 분석⋅전략⋅의사결정 활용 측면에서 데이터를 봄
- 즉, CDO가 데이터 기반과 제공 체계를 더 본다면, CAO는 그 데이터를 어떻게 해석하고 의사결정에 연결할지에 더 가까움

🔹 최고 알고리즘 책임자(CAO-2)

- CAO-2는 ML·AI 연구와 알고리즘 리더십에 집중하는 고도로 전문화된 역할
- 일반적인 주니어 데이터 엔지니어 실무와는 거리가 있지만, 데이터 조직이 커지면 이런 전문 리더십도 생길 수 있다 정도로 이해하면 됨

🔹 데이터 엔지니어 및 프로젝트 매니저

- 대규모 데이터 프로젝트에서는 프로젝트 매니저가 우선순위, 일정, 이해관계자 조율을 담당
- 데이터 엔지니어는 기술 구현에 집중하되, 진척 상황과 장애 요인을 명확히 공유하는 협업 능력이 중요

🔹 데이터 엔지니어와 제품 관리자

- 제품 관리자는 데이터 제품이 어떤 사용자 문제를 해결해야 하는지를 정의하고 조율
- 그래서 데이터 엔지니어도 점점 파이프라인 중심 사고만이 아니라 제품 관점을 함께 가져야 함

🔹 데이터 엔지니어와 기타 관리 역할

- 데이터 엔지니어는 여러 관리자와 협업하며, 조직에 따라 중앙 데이터 팀으로 일할 수도 있고 특정 프로젝트⋅제품 팀에 배치될 수도 있음
- 따라서 주니어는 기술뿐 아니라, 조직 구조 안에서 어떻게 협업하는지 이해하는 감각도 필요
